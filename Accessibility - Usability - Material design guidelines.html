无障碍

可访问的设计使所有能力的用户能够成功地导航，理解和使用您的UI。

原则

所有能力的用户都可以使用精心设计的产品，包括视力低下，失明，听力障碍，认知障碍或运动障碍的用户。 提高产品的可访问性可以提高所有用户的可用性。 这也是正确的事情。

材料设计的内置辅助功能考虑将有助于您适应所有用户。 本节主要适用于移动UI设计。 有关设计和开发完全可访问产品的更多信息，请访问Google辅助功能网站。

颜色和对比度

使用颜色和对比度来帮助用户查看和解释您的应用的内容，与正确的元素交互，并了解操作。

无障碍调色板

为您的应用选择支持可用性的主要，辅助和重音颜色。 确保元素之间的足够的颜色对比度，以便具有低视力的用户可以看到并使用您的应用程序。

对比度

根据万维网联盟（W3C），颜色和背景之间的对比度比例根据其亮度或发光强度的范围为1-21。

对比度表示颜色与另一种颜色的差异，通常写为1：1或21：1。 这两个数字之间的差值越大，颜色之间的相对亮度差越大。

W3C建议对身体文字和图像文字的对比度如下：

  • 与其背景相比，小文本的对比度应至少为4.5：1。
  • 大文本（14点/ 18点常规和以上）应该具有至少3：1的对比度与其背景。

图标或其他关键元素也应使用上述推荐的对比度。

徽标和装饰元素

虽然装饰元素（如徽标或插图）不必满足对比度，但如果它们具有重要的功能，它们应该是可区分的。

其他视觉线索

对于彩色盲人或看不到色差的用户，除了色彩之外，还包括设计元素，以确保他们收到相同的信息量。

色盲采取不同的形式（包括红 - 绿，蓝 - 黄和单色）。 使用多个视觉线索来传达重要的状态。 使用诸如笔画，指示符，图案，纹理或文字的元素来描述动作和内容。

声音和动作

声音

给出声音的视觉替代品，反之亦然。 提供隐藏的字幕，抄本或其他视觉替代关键音频元素和声音警报。

允许用户使用声音导航您的应用程序，为UI元素添加描述性标签。 当使用诸如“话语提示”的屏幕阅读器并通过触摸探索进行导航时，当用户用指尖触摸UI元素时，会大声朗读标签。

应避免以下声音：

  • 在屏幕阅读器上播放不必要的声音，例如进入网页时自动播放的背景音乐。 如果有背景声音，请确保用户可以安全地暂停或停止。
  • 额外的声音添加到本机元素（屏幕阅读器将能够正确解释本机元素）。

Add audible descriptions to input controls and other elements

运动

材料设计使用运动来指导视图之间的焦点，这有助于减少用户分心。 曲面转换为焦点以供用户跟随，不重要的元素被删除。

为了让具有运动和视觉灵敏度的用户能够舒适地使用界面，请遵守材料设计运动规范，其中支持以下运动指南：W3C：

  • 启用自动移动，滚动或闪烁的内容，如果持续超过五秒钟，则暂停，停止或隐藏。
  • 在一秒钟内将闪光内容限制在三次，以满足闪光灯和红色闪光阈值。
  • 避免闪烁屏幕的大中央区域。

定时控制

应用程序中的控件可能会在一定时间后设置为消失。 例如，启动视频后5秒，播放控制可能会从屏幕上淡出。

高优先级控制
避免在执行高优先级功能的控件上使用定时器，因为用户可能不会注意到这些控件是否消失得太快。 例如，话语提示重点读取控件，并将它们放在计时器上可能会阻止控件完成任务。

对于启用其他重要功能的控件，请确保用户可以再次打开控件或以其他方式执行相同的功能。 了解更多的层次结构和焦点。

样式

布局

材料设计触摸目标指南可让无法看到屏幕的用户或具有运动灵巧问题的用户点击应用中的元素。

触摸目标

触摸目标是响应用户输入的屏幕部分。 它们超出元素的视觉范围。 例如，一个图标可能看起来是24 x 24 dp，但它周围的填充包含完整的48 x 48 dp触摸目标。

触摸目标应至少为48 x 48 dp。 这种尺寸的触摸目标导致约9mm的物理尺寸，而不管屏幕大小。 触摸屏元件的推荐目标尺寸为7-10mm。 使用较大的触摸目标来适应更大范围的用户，例如具有发展运动技能的儿童可能是合适的。

触摸目标间距

在大多数情况下，触摸目标应分开8dp的空间或更多，以确保平衡的信息密度和可用性。

分组项目

保持相关项目彼此接近有助于那些视力低下或可能无法集中在屏幕上的人。

字体

为了提高可读性，用户可能会增加字体大小。 移动设备和浏览器包括允许用户在系统范围内调整字体大小的功能。 要在Android应用程序中启用系统字体大小，请以可扩展的像素（sp）标记文本及其关联的容器。

确保为大型和外来字体分配足够的空间。 有关推荐的外语字体大小的信息，请参阅行高。

层次和重点

应用程序应该向用户提供反馈，并了解应用程序的位置。 导航控件应易于定位和清晰的写入。 视觉反馈（例如标签，颜色和图标）和触摸反馈显示用户在UI中可用的内容。

导航应该有最小的步骤清晰的任务流程。 焦点控制，或控制键盘和阅读焦点的能力，应该为经常使用的任务实施。

屏幕阅读器

屏幕阅读器为用户提供了多种导航屏幕的方法，包括：

  • 触摸界面屏幕阅读器允许用户在屏幕上运行他们的手指来听到直接在下面的内容。 这为用户提供了对整个界面的快速感知。 或者用户可以从肌肉记忆快速移动到UI元素。 在“话语提示”中，此功能称为“触摸探索”。要选择项目，用户必须双击。
  • 用户还可以通过在屏幕上向后或向后滑动来移动焦点，从上到下线性地读取页面。 这允许用户磨练某些元素。 在TalkBack中，这被称为线性导航。

用户可以在“触摸探索”和“线性导航”模式之间切换。 当这些地标使用适当的语义标记时，一些辅助技术允许用户在页面地标（例如标题）之间导航。

硬件或软件方向控制器（例如D-pad，轨迹球或键盘）允许用户以线性方式从选择跳转到选择。

等级制度

根据屏幕的相对重要程度将项目放在屏幕上。

  • 重要操作： 将重要操作放置在屏幕顶部或底部（可通过快捷方式访问）。
  • 相关项目：将类似层次结构的相关项目放在彼此之间。

焦点订单

输入焦点应遵循视觉布局的顺序，从屏幕的顶部到底部。 它应该从最重要的到最不重要的项目。 确定以下焦点和动作：

  • 元素接收焦点的顺序
  • 元素分组的方式
  • 当焦点中的元素消失时，焦点移动的位置

通过视觉指标和可访问性文本的组合来澄清焦点存在的位置。

分组

在标题下组合类似的项目来传达分组。 这些组织在空间上组织内容。

转变

屏幕和任务之间的对焦遍历应尽可能连续。

如果任务中断，然后恢复，请将焦点放在先前关注的元素上。

履行

通过使用标准平台控件，您的应用程序将自动包含使用平台辅助技术良好运行所需的标记和代码。 调整您的应用程序，以满足每个平台的可访问性标准和辅助技术（包括快捷方式和结构），为用户提供高效的体验。

在平台可访问性设置打开（在实施期间和之后）测试您的设计。

其他设计考虑：

  • 使用可扩展的文本和宽敞的布局 以适应可能具有大文本，颜色校正，放大或其他辅助设置的用户。
  • 键盘/鼠标接口 应该具有每个任务，所有悬停信息只能通过键盘访问。
  • 触摸界面 应允许屏幕阅读器和其他辅助技术设备读取界面的所有部分。 朗读的文本应该是有意义和有益的。

标注可视化UI元素

屏幕阅读器用户需要知道哪些UI元素可以在屏幕上显示。 要使屏幕阅读器能够大声读取组件名称，请将contentDescription属性添加到包含没有可见文本图标的按钮，图标和选项卡等组件。

Labeling UI Elements
Use the contentDescription attribute to set labels.

帮助文档

任何具有特殊辅助功能要素的功能都应包含在帮助文档中。 使帮助文档相关，可访问和可发现。 例如，请查看本指南，了解如何在Google云端硬盘中使用屏幕阅读器。

测试和研究

遵循这些无障碍指南有助于提高应用程序的可访问性，但不能保证完全可访问的体验。 建议您也：

  • 测试您的应用程序完成任务完成，开始结束，启用各种辅助技术。 例如，在“话语提示”中打开“触摸浏览”，并更改大声朗读文本的速度。
  • 让有损伤的用户测试您的应用程序。
  • 考虑如何可以访问单个元素，同时将它们融合在一起的一致的用户流中。
  • 确保您希望用户完成的主要任务对所有人都是可能的。

与您的用户，特别是那些使用辅助技术的用户，了解他们的需求，他们想要的应用程序，他们使用哪些工具，以及如何使用它们。 熟悉这些工具，以便您可以给他们最好的体验。

写作

清晰有用的辅助功能文字是使UI更易于访问的主要方式之一。 有限或无视力的用户受益于明确的口头描述。 辅助功能文字是指屏幕阅读器可访问性软件使用的文字，例如Android上的TalkBack，iOS上的VoiceOver和桌面上的JAWS。 屏幕阅读器朗读屏幕上的所有文本，包括可见和不可见的替代文本。

辅助功能文本包括可见文本（包括UI元素的标签，按钮上的文本，链接和表单）以及不显示在屏幕上的不可见描述（例如，没有文本标签的按钮的替代文本）。 有时候，屏幕标签可能会覆盖无障碍文字，以便为用户提供更多信息。

可见和不可见的文本应该是有帮助的描述性和独立的意义，因为一些用户通过使用页面上的所有标题或链接进行导航。 使用屏幕阅读器测试您的应用程序，以识别缺少的区域或需要更好的辅助功能文本。

简洁

要保持内容和辅助功能文本的简短。 屏幕阅读器用户听到每个UI元素朗读。 文本越短，屏幕阅读器用户可以浏览的速度越快。

开发人员注意： 如果控件类型或状态未正确读取，则控件的可访问性角色可能设置不当或是自定义控件。 每个元素应该在网站上具有相关的可访问性角色，或者被编码以便正确地宣布。 这意味着应该将按钮设置为按钮，并将复选框设置为复选框，以使控件的类型或状态正确地传达给用户。 如果您从本机UI元素扩展或继承，您将获得正确的角色。 如果没有，您可以覆盖每个平台上的辅助功能信息（ARIA for web，AccessibilityNodeInfo for Android）。

在Android上，将控件的AccessibilityNodeInfo的类名字段设置为“android.widget.Button”。

指示元素的作用

使用动作动词表示元素或链接的作用，而不是元素的外观，因此视力障碍者可以理解。 链接文本应该：

  • 指定点击链接将执行的任务
  • 避免含糊的描述，如“点击这里”

确保一个元素在使用的地方有相同的描述。

具有状态变化的元素

对于在值或状态之间切换的图标，可以根据用户的显示方式通知图标。

  • 如果图标是项目的属性，请使其成为复选框，以便屏幕阅读器对当前状态进行说明，例如“开”或“关”。
  • 如果图标是一个动作，则写入文本标签以指定如果图标被选择时发生的操作，例如“添加到愿望清单”。

元素如何被使用会影响它们的显示方式。 例如，如果星形图标表示将某些内容添加到愿望清单中的操作，应用程序应该将“添加到愿望清单”或“从愿望清单中删除”进行说明。

不要提到确切的手势或互动

不要告诉用户如何与控件进行物理交互，因为它们可能是使用键盘或其他设备进行导航，而不是用手指或鼠标。 辅助功能软件将描述用户的正确交互。

确认操作

使用对话框，toast包机或snackbars（Android）来确认或确认具有破坏性的用户操作（如“删除”或“删除”）或难以撤消。

对于通过视觉手段确认的操作，例如，当项目被删除时，网格重新排列，则不需要吐司。 在这些情况下，添加辅助功能文本以提供确认。

提供提示讲话

提示语言为不清楚的动作提供额外的信息。 例如，Android的“双击选择”功能会提示用户在登陆物品时点击两次而不采取行动。 Android TalkBack还将宣布与元素相关联的任何自定义操作。 谨慎使用提示语言，仅适用于复杂的用户界面。
